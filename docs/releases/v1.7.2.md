**发布日期**: 2025-04-25

## 🎯 发布概述

PlookingII v1.7.2 是一个**关键性能修复版本**，彻底解决了生产环境中长期运行后性能雪崩的问题。经过深入分析和修复，应用现在可以在浏览大量高分辨率照片时保持稳定的性能和流畅的切换体验。

**核心修复**：

- 🚨 修复缓存内存记账使用文件大小导致的 LRU 失效
- 🩺 修复所有内存清理逻辑（之前引用不存在属性，从未生效）
- 💉 消除 ImageIO 双重缓存，减少 50% 内存消耗
- ⚡ 限制后台线程并发，消除导航卡顿

## 🐛 关键 Bug 修复

### 🚨 1. 内存泄漏修复（性能雪崩根因）

#### 问题分析

在 v1.7.1 中，缓存系统使用**文件大小**而非**解码后的像素内存大小**来记账：

```
一张 6000×4000 照片:
  文件大小: 5MB    ← 实际记账
  解码后像素: 96MB  ← 实际占用的内存
```

由于记账仅为实际的 5%，缓存认为远未触及 500MB 上限，**LRU 淘汰策略永远不会触发**。同时，内存清理代码引用了不存在的属性（如 `preview_cache`），**所有清理逻辑静默失效**。这两种缺陷叠加，导致：

1. 缓存无限制增长到数十 GB（系统开始使用交换）
2. 后台线程大量堆积（每次导航创建 7-8 个线程）
3. macOS 内存压力高 → 系统级节流
4. UI 卡顿 → 几乎不可用

#### 修复方案

| 修复项 | 之前 | 之后 |
|--------|------|------|
| 缓存记账 | 文件大小（5MB） | 像素大小（96MB） |
| LRU 触发 | 永不触发 | 20 项 ≈ 1920MB 自动淘汰 |
| 内存清理 | 引用不存在属性（静默吞异常） | `evict_oldest()` + `gc.collect()` |
| 缓存上限 | 50 项 / 500MB（基于错误记账） | 20 项 / 2000MB（基于正确记账） |
| ImageIO 缓存 | `kCGImageSourceShouldCache=True` | `False`（消除双重缓存） |
| 线程创建 | 每次导航 7-8 个 `threading.Thread` | `ThreadPoolExecutor(max_workers=4)` |

#### 影响

- ✅ 修复前：浏览 50 张大图后开始卡顿，100 张后几乎不可用
- ✅ 修复后：稳定运行，内存保持在 1.5-2.0GB 范围，切换流畅

## 🔧 改进项

### 1. API 兼容性

- 新增 `SimpleImageCache.evict_oldest(count)` 公开方法
- 新增 `SimpleImageCache.get(key, target_size=None)` 添加 `target_size` 形参（兼容外部调用）
- 新增模块级 `estimate_image_memory_mb(image)` 函数
- ✅ 所有现有 API 保持不变

### 2. 代码质量

- 删除所有对 `put_new`、`_evict_oldest_preview`、`_evict_oldest_preload`、`get_size` 等不存在方法的引用
- 模块级函数 `estimate_image_memory_mb` 统一管理内存估算逻辑
- 统一使用 `SimpleImageCache.put(key, value, size_mb)` 签名
